# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[convention:project]** A native module spanning all 3 layers (e.g. PhlixWebAuthn) MUST keep method names + arg order + the JSON-string-in / JSON-string-out shape IDENTICAL across `src/native/types.ts`, the iOS Swift `@objc(...)` module + its `.m` `RCT_EXTERN_METHOD` bridge, and the Android Kotlin `@ReactMethod` module. State this contract in a header comment in each file so they stay in lockstep.
- **[pattern:project]** Wiring a NEW local native module: Android needs BOTH a manual `add(XxxPackage())` in `MainApplication.kt` `getPackages()` AND the dependency in `android/app/build.gradle` (local modules are NOT autolinked); iOS LocalPod `PhlixPlayer.podspec` `s.source_files = "*.{h,m,swift}"` auto-globs new `.swift`/`.m` files but you must add any system framework to `s.frameworks` (e.g. `"AuthenticationServices"`) and re-run `pod install`.
- **[pattern:project]** Device-only native features (passkeys/WebAuthn — no simulator/CI can run the system biometric sheet) must gate ALL UI behind an async `isSupported()` probe that resolves false when the native module is absent. This keeps Jest + non-native runs functional; mirror it (probe in a `useEffect`, hide the affordance) on every screen that uses the module.
- **[gotcha:project]** `grep -c pattern file && nextcmd` aborts the rest of the chain: `grep -c` EXITS 1 when the count is 0 (returning `0`), which short-circuits `&&` and the combined Bash call returns empty/looks failed. Split the probe into its own command, or append `|| true`.
- **[gotcha:project]** When a subagent authors a pure helper AND its unit test together (it can't run jest), they can encode contradictory edge-case assumptions that only surface when the top-level runs the FULL `npm test` — treat the failing test as the intent spec. Seen twice: `clampRating` Infinity handling, and a `webauthnErrorMessage` matcher that checked `'not allowed'` but the real error string was the concatenated `'NotAllowedError'` (fix: also match `'notallowed'`). Error-string matchers must cover BOTH spaced and concatenated forms. Always run the whole suite after a subagent ships; never trust its self-reported pass.
- **[env:project]** To compile-check the iOS Swift native modules WITHOUT macOS/Xcode (Linux dev box), run `/home/my/.local/share/swiftly/bin/swift -frontend -parse <file>.swift` and filter out `warning:` lines — `swift` is at `/home/my/.local/share/swiftly/bin/swift`, not on `$PATH`. The full pre-push gate for a native-touching slice is `npm run typecheck && npm run lint && swift -frontend -parse <file>` (jest separately); the passkey RUNTIME stays device-only-unverifiable.
- **[gotcha:project]** `UIApplication.shared.windows` is DEPRECATED since iOS 15 — for an `ASPresentationAnchor` (passkey/AuthenticationServices presentation) walk `UIApplication.shared.connectedScenes`, filter `UIWindowScene` with `.activationState == .foregroundActive`, then `.windows.first { $0.isKeyWindow }`. Multi-scene / iPad safe.
- **[pattern:project]** When a native ceremony returns a JSON string (PhlixWebAuthn register/authenticate), wrap the `JSON.parse` in a try/catch that rethrows a clear domain error (e.g. "Passkey ceremony returned an invalid response.") instead of letting a raw `SyntaxError` leak out of the service — a malformed/interrupted native payload should fail with an actionable message.
- **[gotcha:project]** When a subagent (which cannot run jest) authors a pure helper AND its unit test together, they can encode contradictory edge-case assumptions that only surface when the top-level runs the FULL `npm test` — treat the failing test as the intent spec. Seen: `clampRating` Infinity handling, and a `webauthnErrorMessage` matcher checking `'not allowed'` when the real string was concatenated `'NotAllowedError'` (fix: match BOTH spaced and concatenated forms). Always run the whole suite after a subagent ships; never trust its self-reported pass.
- **[pattern:project]** Manager methods that take optional query params build an EXPLICIT params object: REQUIRED keys always present and first, only truly-optional keys omitted when undefined — so a no-arg call sends just the required keys (or `{}`/`undefined` params entirely, e.g. `registerOptions()` body `{}` when no username, `tailAllLogs()`→`get(url, undefined)`). Assert both the minimal and populated params in the manager test.
- **[gotcha:project]** Music/photo/favorites JSON routes are **Bearer-gated, NOT admin-gated** (unlike Profiles/Admin/LiveTv) — do not wrap them behind an `is_admin` gate. WebAuthn has TWO auth contexts: register/list/delete are BEARER, but `login/options`+`login/verify` are PRE-AUTH (passwordless) and the verify step RETURNS the token envelope. `/api/v1/photo/*` is SINGULAR. Photo `thumbnail_url`/`full_url` payload fields are SIGNED → feed straight into `<Image source={{ uri }}>` with NO Bearer header.
- **[convention:project]** Do NOT pass arrays/objects through React Navigation params — pass only serializable scalars (ids + an index) and RE-FETCH the data in the destination screen (PhotoAlbum→PhotoViewer; CollectionDetail re-fetches by id). Mirror this for any list→detail/viewer navigation.
- **[convention:project]** Admin/privileged screens gate on `!!useAuthStore.user?.is_admin`, are registered in the ROOT stack (not the tab bar), reached via a Settings section rendered ONLY under `user?.is_admin`, and return the "Admin access required" EmptyState BEFORE any data-loading effect fires. Screens needing route data take PARAMS typed in `RootStackParamList` and are registered as headered root-stack screens.

- **[pattern:project]** Cumulative-offset jump rails (letter rail): the letter-index `offset` is a server `?offset=` value, NOT a list index — feed it into `browseMedia({...,offset})`, REPLACE results, keep ONE monotonic cursor; build both the browse and index calls from the SAME `buildBrowseParams(query, filters)`; show the rail only in browse mode (no text query) + name sort + results>0.
- **[gotcha:project]** `swift -frontend -parse <file>.swift` (the Linux compile-check) does NOT catch semantic/override errors — it only parses syntax. CI caught `override static func requiresMainQueueSetup()` failing with "method does not override any method from its superclass" that the local parse passed clean. A LocalPod Swift module subclassing `NSObject` directly (plain `RCT_EXTERN_MODULE`, NOT `RCTViewManager`/`RCTEventEmitter`) must declare `@objc static func requiresMainQueueSetup() -> Bool` — NOT `override` (only `RCTViewManager`/`RCTEventEmitter` subclasses have it to override). Real semantic verification still requires Xcode/CI.
- **[gotcha:project]** In an Android `ReactContextBaseJavaModule`, `currentActivity` is NOT in scope as a bare reference — it resolves via `reactApplicationContext.currentActivity` (returns `Activity?`). A bare `currentActivity` fails `compileDebugKotlin` with `Unresolved reference 'currentActivity'` + a downstream `Argument type mismatch: actual type is 'Any', but 'Context' was expected`. There is no Linux Kotlin gate either — these surface only in the Android CI build.
- **[pattern:project]** Native (iOS Swift / Android Kotlin) modules in this repo are NOT covered by typecheck/lint/jest/swift-parse — the ONLY full compile check is the GitHub `build-ios.yml` / `build-android.yml` workflows. After pushing a native-touching branch, watch BOTH builds and pull errors with `gh run list --branch <b> --workflow build-ios.yml --limit 1 --json databaseId --jq '.[0].databaseId'` then `gh run view <id> --log-failed | grep -iE "error: |\.swift:|\.kt:|e: file://"` (filter out `warning:`/`export ` noise). Expect to fix-and-repush at least once for a new native module — the local gates give false confidence.

A `Codacy Static Code Analysis` fail is independent of build/test, NOT a build break.

- **[gotcha:project]** GitHub Actions upload-artifact ALWAYS serves artifacts zipped — there is no way to offer a bare, directly-installable `app-release.apk` from an artifact. To publish a download-and-install APK, add a `release-latest` job (after `build-android-release`) that downloads the artifact and re-publishes it as a GitHub Release asset via `softprops/action-gh-release`. Make the "latest" release a rolling overwrite: `gh release delete latest --yes --cleanup-tag || true` first, then re-create with `tag_name: latest`, `prerelease: true`, `make_latest: false`, `target_commitish: ${{ github.sha }}`. Gate the job to push events on master/main only (`if: github.event_name == 'push' && (github.ref == 'refs/heads/master' || ...)`) and grant `permissions: contents: write`.
- **[env:project]** CI iOS builds run WITHOUT Apple code signing (`CODE_SIGNING_ALLOWED=NO`) — they compile-only and cannot produce an installable/distributable IPA. Do NOT try to publish an iOS artifact as a downloadable release asset; only the Android release APK is publishable from CI. App Store / device installs need Apple signing that CI does not have.
- **[gotcha:project]** An Android `release` buildType with NO `signingConfig` produces an UNSIGNED `app-release.apk` that cannot be sideloaded/installed. For the rolling-`latest` sideload artifact, set `signingConfig signingConfigs.debug` in `android/app/build.gradle`'s `buildTypes.release {}` so `assembleRelease` emits an installable APK; a real Play Store build should swap in a proper release keystore (CI secrets, on a version tag). This is separate from — and required IN ADDITION to — the CI release-publishing job.
